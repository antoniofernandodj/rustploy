# Status do Projeto — 26 de maio de 2026

## Visão geral

O Rustploy é um PaaS mínimo em Rust, composto por:
- **`rustployd`** (daemon): servidor HTTP sobre Unix Domain Socket; gerencia projetos,
  serviços, deployments, Docker e o proxy de ingresso.
- **`rustploy`** (client): TUI Ratatui que se comunica com o daemon via RPC postcard.
- **`shared`**: tipos de protocolo, modelos e config compartilhados.

O pipeline de deploy Git → Docker está **funcionando ponta a ponta**. O container
`rp_teste` está marcado como **RUNNING** no TUI e executando de fato no Docker.

## O que foi resolvido (leitura do `git diff`)

### 1. Deploy Git ficava eternamente preso em `Deploying` — `commit dbcdac6`

**Causa raiz:** `db/deployments::transition` usava `WHERE id = $id` numa query
SurrealDB. O campo `id` é do tipo `Thing` (SurrealDB record ID), mas a variável
`$id` era uma `String` simples. A comparação always falhava silenciosamente e o
executor lia o deployment com o estado anterior, nunca avançava, entrava em loop
infinito e o serviço ficava eternamente `Deploying`.

**Fix:** Trocar para `UPDATE type::thing('deployment', $id) SET …` — endereça o
registro diretamente pelo record ID tipado.

---

### 2. `git2` rejeitava diretório de clone não-vazio — `commit dbcdac6`

**Causa:** Em re-deploys o diretório `builds/<deployment_id>/` já existia (de um
deploy anterior com o mesmo ID, ou por sujeira). O `git2::RepoBuilder::clone`
exige diretório vazio e falha com `exists and is not an empty directory`.

**Fix:** Limpar o diretório antes de clonar via `std::fs::remove_dir_all`.

---

### 3. Clone `file://` não resolvia branches corretamente — unstaged

**Causa:** O `git2` com URL `file://` faz clone por hardlink (clone local). Isso
não popula `refs/remotes/origin/<branch>`, então `builder.branch("main")` falha.
Também, o próprio método `.branch()` do `RepoBuilder` já chamava internamente a
resolução de `refs/remotes/origin/main` que não existia.

**Fix (unstaged):**
- Remover o `builder.branch()` do `RepoBuilder` e fazer checkout manual após o clone.
- Strip do prefixo `file://` para forçar o caminho absoluto (mais confiável no git2).
- Resolução com fallback: tenta `origin/<branch>` primeiro (clones de rede), depois
  `<branch>` diretamente (clones locais/file://).

---

### 4. IP do container vinha vazio em redes user-defined — unstaged

**Causa:** `bollard::EndpointSettings.ip_address` fica vazio quando a conexão de
rede é feita via `network_mode` no `HostConfig.create` para redes user-defined.
O Docker só preenche `NetworkSettings.Networks.<name>.IPAddress` se a conexão for
via `docker network connect` após o create.

**Fix (unstaged) — dois sub-fixes interdependentes:**

a) **`create_staging`**: removido `network_mode` do `HostConfig`. O container é
   criado sem rede user-defined, conectado explicitamente via
   `networks::connect_container()` **antes** do `containers::start()`.
   Sequência: `create → network connect → start`.

b) **`get_container_ip`**: trocado de `docker container inspect` (lê
   `EndpointSettings.ip_address`) para `docker network inspect` (lê
   `NetworkContainer.ipv4_address`). A segunda struct é populada corretamente.
   O endereço vem no formato CIDR `"172.19.0.3/16"` — extraído só o IP com `.split('/')`.

---

### 5. Healthcheck `DockerNative` em imagem sem `HEALTHCHECK` bloqueava forever — unstaged

**Causa:** A condição de sucesso era `status == Some(&HealthStatusEnum::HEALTHY)`.
Quando a imagem não define `HEALTHCHECK`, o Docker retorna `status = None` (campo
ausente no JSON). O polling nunca passou.

**Fix (unstaged):** Tratar `None` como sucesso (container rodando = saudável o
suficiente). Só bloqueia em `UNHEALTHY` explícito. `STARTING` aguarda.

---

### 6. Caminho do master key requeria root — unstaged

**Causa:** O caminho padrão `/etc/rustploy/master.key` exige root para criar.
Sem root o `SecretsManager::new()` falhava na inicialização.

**Fix (unstaged):** `resolve_master_key_path()` — mesma lógica já existente para
o DB path: tenta o caminho configurado, faz fallback para
`~/.local/share/rustploy/master.key` se o diretório não for gravável.

---

### 7. `.git/` incluído no tar do build context — `commit dbcdac6`

**Causa:** `create_tar_gz` incluía o diretório `.git/` no contexto enviado ao
Docker build, inflando desnecessariamente o contexto e podendo vazar credenciais.

**Fix:** Filtrar entradas com prefixo `.git` no walker do build context.

---

### 8. `ServiceUpdate` corrompendo `ServiceSource` — `commit dbcdac6`

**Causa:** `db/services:
:update()` usava o método `.merge()` do SurrealDB SDK.
Para enums Rust serializados como tagged unions JSON, o merge fazia deep-merge dos
campos e corrompía o enum (ex.: `Git` virava um objeto com campos misturados de
`Registry`).

**Fix:** Substituir `.merge()` por `UPDATE … SET campo = $valor` com binds explícitos
para cada campo da spec.

---

### 9. TUI: fluxo de criação de serviço multi-passo — `commit da53bcf`

Substituída a tela única `ServiceForm` por um popup de 4 passos estilo Dokploy:
`PickType → PickDbType → ApplicationForm | DatabaseForm`.

Suporte a `DbKind`: Postgres, MongoDB, MariaDB, MySQL, Redis com imagens, portas
e env vars pré-preenchidos. O formulário de DB suporta scroll para campos extras.

### 10. Aba Healthcheck na tela de serviço — unstaged (trabalho em andamento)

## Problemas pendentes (ainda não resolvidos)

### 🔴 Logs do container não streamam para a aba Logs

**Sintoma:** Container está `Running`, mas a aba "Logs" do TUI mostra
"Aguardando logs... (serviço precisa estar Running)".

**Causa:** O daemon **não tem nenhum background task** que faça `docker logs --follow
<container>` e publique `Event::LogLine` no event bus para containers vivos.

A única publicação de `Event::LogLine` que existe hoje é no build da imagem
(`crates/daemon/src/docker/images.rs:106`), que publica as linhas de output do
`docker build`. Após o container entrar em `Live`, nada lê o stdout/stderr.

**O que precisa ser implementado:**
- Uma task tokio (`tokio::spawn`) que, ao container entrar em `Live`, assina
  `docker.logs(container_id, LogsOptions { follow: true, stdout: true, stderr: true, … })`
  e publica cada linha como `Event::LogLine { service_id, … }`.
- A task precisa ser cancelada/substituída em redeploys (quando o container live muda).
- Candidato para um `CancellationToken` por serviço, guardado em um `HashMap` no
  estado global do daemon.

### 🟡 Stop do container antigo demora 30s

O container do deploy anterior (`e04823fdddea`) ignorou SIGTERM e levou 30s para
ser parado (Docker timeout → SIGKILL). Isso é comportamento do container de teste
`hw` (um hello-world em loop sem handler de sinal), mas aumenta o tempo total de
deploy em ~30s.

Para produção, considerar reduzir o timeout de `stop_graceful` de 30s para algo
menor (5-10s) e/ou documentar que serviços devem tratar SIGTERM.
